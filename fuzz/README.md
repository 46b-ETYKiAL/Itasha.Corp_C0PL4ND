# C0PL4ND fuzz harnesses

This is a [`cargo-fuzz`](https://github.com/rust-fuzz/cargo-fuzz) workspace
member. It is **its own workspace** (see the empty `[workspace]` table in
`fuzz/Cargo.toml`) and the parent `Cargo.toml` lists it under
`[workspace].exclude`, so the unstable, sanitizer-instrumented fuzz build never
pollutes the stable `cargo build` / `cargo test --workspace` matrix. This is the
layout the [Rust Fuzz Book](https://rust-fuzz.github.io/book/) prescribes.

Fuzzing requires a **nightly** toolchain (libFuzzer needs the sanitizer-
instrumented build) plus the `cargo-fuzz` subcommand:

```sh
cargo install cargo-fuzz
```

## Targets

| Target | Entry point (all public on `c0pl4nd_core`) | Untrusted-input surface |
|--------|--------------------------------------------|--------------------------|
| `vt_parser`  | `c0pl4nd_core::Terminal::advance` | VT / ANSI / OSC escape-sequence bytes — produced by **any** program running inside the terminal, so fully untrusted. |
| `state_json` | `c0pl4nd_core::layout_persist::{WorkspaceSnapshot, LayoutSnapshot}::from_json` (+ `validate` / `restore` / `restore_all`) | Persisted window/pane/workspace layout JSON — untrusted **at rest** (hand-edited, crash-truncated, or tampered config file). |

### `vt_parser`

Feeds fuzzer-generated bytes into `Terminal::advance` in arbitrarily-sized
chunks (the first input byte selects a 1..=64 chunk size), so escape sequences
that straddle PTY reads are exercised — the realistic case where a sequence is
split across reads. After parsing it reads back the public derived state
(`title`, `cwd`, `prompt_marks`, `hyperlinks`, `images`, `scrollback_len`,
`grid`, `display_rows`, `all_lines`) so any inconsistency reachable only through
derived state is caught. A malformed or hostile escape stream must never panic,
hang, OOB, or corrupt the grid model.

### `state_json`

Drives both persisted-state parse entry points: the multi-tab
`WorkspaceSnapshot::from_json` (which includes the v1 → v2 migration shim and
the unsupported-version reject branch) and the bare single-tab
`LayoutSnapshot::from_json`. On any input that parses it also runs the
reconstruction paths (`restore_all` / `restore`), which walk the layout tree,
allocate ids, count leaves, and enforce the `MAX_PANES` cap — the arithmetic /
recursion logic most likely to hit an edge on a hostile tree shape. The loader
is documented to *fall back* on bad input; this target asserts it never aborts
instead.

## Running

```sh
# Run a single target (Ctrl-C to stop; corpus + crashes land under fuzz/).
cargo +nightly fuzz run vt_parser
cargo +nightly fuzz run state_json

# Time-box a run (e.g. 60s) — handy in a smoke loop.
cargo +nightly fuzz run vt_parser -- -max_total_time=60

# Just compile every target without fuzzing (the CI bitrot guard does this).
cargo +nightly fuzz build

# List the registered targets.
cargo +nightly fuzz list
```

CI compiles the targets on nightly (`cargo fuzz build`) on every push/PR so they
cannot bitrot, but does **not** run long fuzzing campaigns in CI — fuzzing is an
unbounded search and belongs in a dedicated/scheduled run, not the per-PR gate.

## Known limitation — the updater archive extractor is not fuzzed here

The updater's archive extractors
(`extract_binary` / `extract_binary_zip` / `extract_binary_targz` in
`crates/app/src/update_engine/net.rs`) are an untrusted-input surface worth
fuzzing — they decode network-fetched `.zip` / `.tar.gz` bytes (`flate2` + `tar`
+ `zip`) into a temp dir — but they are **not reachable from this fuzz crate**
for two structural reasons:

1. **`crates/app` is a binary-only crate** (no `lib.rs`; it builds the `c0pl4nd`
   and `c0pl4nd-legacy` binaries). A `cargo-fuzz` target is itself a crate and
   can only depend on a *library* crate, so it cannot link against the app's
   internals at all.
2. **The extractor functions are private** (`fn`, not `pub fn`). Even with a
   library facade they would be unreachable without source edits.

Exposing them would require editing `crates/**` (adding a `lib.rs` to
`crates/app` and/or widening visibility), which is **out of scope** for this
change set. The mitigations already in place on that surface make the gap
low-risk in the interim:

- The extractor enforces a **max entry count** (`MAX_ARCHIVE_ENTRIES`) and a
  **max decompressed size** (zip-bomb / tar-bomb guard) before writing anything.
- Downloaded bytes are **SHA-256-then-minisign verified** against a build-time
  embedded ed25519 key; an unsigned or tampered archive is rejected
  fail-closed, so extraction only runs on bytes whose integrity was already
  proven.
- The extractor decode logic itself is covered by unit tests in
  `update_engine/net.rs` (round-trip extract, too-many-entries refusal,
  no-binary-entry error, bomb-cap).

To close the gap properly, a future change owning `crates/app` should add a
thin library target exposing the extractor (e.g. a `#[doc(hidden)] pub fn` or a
`fuzzing` cfg gate) and an `updater_archive` fuzz target here that feeds
`arbitrary`-generated archive bytes through it. That work is tracked as a source
change to `crates/app`, not to this fuzz crate.
