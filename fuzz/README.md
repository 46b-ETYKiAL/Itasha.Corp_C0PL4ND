# C0PL4ND fuzz harnesses

This is a [`cargo-fuzz`](https://github.com/rust-fuzz/cargo-fuzz) workspace
member. It is **its own workspace** (see the empty `[workspace]` table in
`fuzz/Cargo.toml`) and the parent `Cargo.toml` lists it under
`[workspace].exclude`, so the unstable, sanitizer-instrumented fuzz build never
pollutes the stable `cargo build` / `cargo test --workspace` matrix. This is the
layout the [Rust Fuzz Book](https://rust-fuzz.github.io/book/) prescribes.

Fuzzing requires a **nightly** toolchain (libFuzzer needs the sanitizer-
instrumented build) plus the `cargo-fuzz` subcommand:

```sh
cargo install cargo-fuzz
```

## Targets

| Target | Entry point | Untrusted-input surface |
|--------|-------------|--------------------------|
| `vt_parser`  | `c0pl4nd_core::Terminal::advance` | VT / ANSI / OSC escape-sequence bytes — produced by **any** program running inside the terminal, so fully untrusted. |
| `state_json` | `c0pl4nd_core::layout_persist::{WorkspaceSnapshot, LayoutSnapshot}::from_json` (+ `validate` / `restore` / `restore_all`) | Persisted window/pane/workspace layout JSON — untrusted **at rest** (hand-edited, crash-truncated, or tampered config file). |
| `fuzz_sixel` | `c0pl4nd_core::image::decode_sixel` | Sixel inline-image bytes — a hand-written DCS-payload byte state-machine, arriving over the PTY from **any** program. |
| `fuzz_kitty` | `c0pl4nd_core::image::{parse_kitty, decode_kitty}` (f=24 / f=32 / f=100) | Kitty graphics protocol control-string + per-format pixel decoder (incl. the PNG path), arriving over the PTY from **any** program. |
| `fuzz_archive` | self-contained extraction **shim** over `zip` / `tar` / `flate2` (see below) | Update-engine archive bytes — `.zip` / `.tar.gz` decompression SHAPE (zip-bomb, malformed headers, path-traversal entry names, truncated streams). |

### `vt_parser`

Feeds fuzzer-generated bytes into `Terminal::advance` in arbitrarily-sized
chunks (the first input byte selects a 1..=64 chunk size), so escape sequences
that straddle PTY reads are exercised — the realistic case where a sequence is
split across reads. After parsing it reads back the public derived state
(`title`, `cwd`, `prompt_marks`, `hyperlinks`, `images`, `scrollback_len`,
`grid`, `display_rows`, `all_lines`) so any inconsistency reachable only through
derived state is caught. A malformed or hostile escape stream must never panic,
hang, OOB, or corrupt the grid model.

### `state_json`

Drives both persisted-state parse entry points: the multi-tab
`WorkspaceSnapshot::from_json` (which includes the v1 → v2 migration shim and
the unsupported-version reject branch) and the bare single-tab
`LayoutSnapshot::from_json`. On any input that parses it also runs the
reconstruction paths (`restore_all` / `restore`), which walk the layout tree,
allocate ids, count leaves, and enforce the `MAX_PANES` cap — the arithmetic /
recursion logic most likely to hit an edge on a hostile tree shape. The loader
is documented to *fall back* on bad input; this target asserts it never aborts
instead.

### `fuzz_sixel`

Feeds arbitrary fuzzer-generated bytes straight into `decode_sixel` — the
hand-written Sixel state-machine (palette parsing, `!N` RLE repeats, 6-pixel
band advance, sparse pixel map, integer arithmetic on attacker-controlled
counts). A malformed or hostile Sixel stream must never panic, hang, overflow,
or OOB; `None` on garbage is correct. After a decode it reads back the image's
`width` / `height` / `rgba.len()` so any internally-inconsistent image is caught.

### `fuzz_kitty`

Drives BOTH Kitty surfaces: `parse_kitty` (the `f`/`s`/`v`/`a`/`m`/`i` control
keys + payload split) and `decode_kitty` for all three supported formats —
`f=24` (RGB→RGBA), `f=32` (RGBA), `f=100` (PNG, handed to the `image` crate).
The leading input bytes derive the format/dimensions so every format path is
exercised across the corpus, and the raw bytes are also fed to the `f=100` PNG
decoder directly. The checked-mul / length-mismatch guards on attacker-controlled
`width`/`height` and the PNG decoder on arbitrary bytes are the logic under test.

### `fuzz_archive`

Drives a **self-contained extraction shim** (defined in the fuzz target) that
faithfully mirrors the production bounded-extraction logic — entry-count cap
(`MAX_ARCHIVE_ENTRIES = 64`), copy-capped inflate (`MAX_EXTRACTED_BYTES`,
saturating), `enclosed_name()` traversal-skip (zip) / `file_name()` selection
(tar), and the `c0pl4nd` / `c0pl4nd.exe` allow-list — over the SAME exact-pinned
`zip` / `tar` / `flate2` crate versions the app uses. It fuzzes the SHAPE of the
decompression path (zip-bomb / tar-bomb / malformed-header / path-traversal
entry-name / truncated-stream classes). See the "archive extractor" note below
for why this is a shim rather than a direct call to the real function.

## Running

```sh
# Run a single target (Ctrl-C to stop; corpus + crashes land under fuzz/).
cargo +nightly fuzz run vt_parser
cargo +nightly fuzz run state_json

# Time-box a run (e.g. 60s) — handy in a smoke loop.
cargo +nightly fuzz run vt_parser -- -max_total_time=60

# Just compile every target without fuzzing (the CI bitrot guard does this).
cargo +nightly fuzz build

# List the registered targets.
cargo +nightly fuzz list
```

CI compiles the targets on nightly (`cargo fuzz build`) on every push/PR so they
cannot bitrot, but does **not** run long fuzzing campaigns in CI — fuzzing is an
unbounded search and belongs in a dedicated/scheduled run, not the per-PR gate.

## The archive extractor — fuzzed via a faithful shim (`fuzz_archive`)

The updater's archive extractors
(`extract_binary` / `extract_binary_zip` / `extract_binary_targz` in
`crates/app/src/update_engine/net.rs`) decode network-fetched `.zip` / `.tar.gz`
bytes (`flate2` + `tar` + `zip`) into a temp dir — a byte-parsing untrusted-input
surface worth fuzzing. They are **not callable directly** from this fuzz crate
for two structural reasons:

1. **`crates/app` is a binary-only crate** (no `lib.rs`; it builds the `c0pl4nd`
   and `c0pl4nd-legacy` binaries). A `cargo-fuzz` target is itself a crate and
   can only depend on a *library* crate, so it cannot link against the app's
   internals at all.
2. **The extractor functions are private** (`fn`, not `pub fn`). Even with a
   library facade they would be unreachable without source edits.

Adding a `lib.rs` to `crates/app` or widening those functions to `pub` is a
source change owned by another stream and out of scope here. So `fuzz_archive`
instead drives a **self-contained shim** that faithfully mirrors the production
bounded-extraction logic against the SAME exact-pinned `zip` / `tar` / `flate2`
crate versions:

- the **max entry count** guard (`MAX_ARCHIVE_ENTRIES = 64`);
- the **copy-capped** inflate (`MAX_EXTRACTED_BYTES`, saturating arithmetic — the
  zip-bomb / tar-bomb guard);
- the zip `enclosed_name()` path-traversal skip + tar `file_name()` selection;
- the `c0pl4nd` / `c0pl4nd.exe` binary-name allow-list.

The shim writes to an in-memory counting sink (not the filesystem) so the inflate
path runs at full fuzzer speed. This fuzzes the SHAPE of the decompression logic
— which is the part that walks attacker-shaped archive structure. If `crates/app`
later grows a `[lib]` target (or marks the extractors `pub`), the shim should be
swapped for a direct call to the real function. The production surface also
retains its existing defences: the extractor only runs on bytes already proven by
**SHA-256-then-minisign verification** (fail-closed), and the decode logic is
covered by unit tests in `update_engine/net.rs` (round-trip extract,
too-many-entries refusal, no-binary-entry error, bomb-cap).
